二、提权处理

1.SUID提权处理

可以用SUID（让程序在执行时，临时拥有文件所有者的权限，而不是当前用户的权限。）来进行提权处理

补:

-perm -4000核心条件：匹配权限

4000 是 SUID 权限的八进制值

-4000 表示至少拥有 SUID 权限（权限位中包含 s）。例如：-rwsr-xr-x（4755）、-rwsr-sr-x（6755）都会被匹配。

2000 → SGID 权限 1000 → Sticky 位

接下来进入/opt/write

因为它是 not stripped，用 IDA 打开时，能直接看到 main 函数和 system、write 等调用，快速定位漏洞。

结合它的 文件写入功能，直接写入 bash -p 提权

2.文件下载

再打开一个终端进行下载

将下载后的write文件放ida进行分析

这是一个SUID 权限特征 的 C 程序

相对路径：system() 函数会调用 /bin/sh -c 来执行命令。由于程序没有指定 warning 的绝对路径（如 /usr/bin/warning），系统会根据当前用户的 $PATH 环境变量 去寻找这个可执行文件。

特权执行：在调用 system 之前，程序执行了 setuid(0)。这意味着 warning 命令将以 root 身份 运行。

劫持过程：

  • 攻击者可以创建一个名为 warning 的恶意脚本（例如：#!/bin/sh\n/bin/sh）。
  • 将该脚本所在的目录添加到自己的 $PATH 最前面。
  • 故意触发程序的错误（例如，通过某种方式让 /opt/welcome.txt 无法打开，或者干脆填满磁盘空间导致写入失败）。
  • 程序执行 system("warning") 时，实际上运行了攻击者的脚本，从而获得一个 root shell。
3.触发程序拿到权限

现在就是要写一个脚本来触发程序执行warning拿到root权限

PATH 环境变量劫持： 这是提权的第一步。Linux 系统在执行没有绝对路径的命令时，会按照 PATH 变量定义的目录顺序逐一查找。通过 export PATH=/tmp:$PATH，成功地将 /tmp 目录置于系统目录（如 /bin）之前。当程序试图调用 warning 时，它会优先执行你存放在 /tmp 下的同名恶意脚本，而不是系统预期的程序。

利用 ulimit 强制触发错误分支： 源码逻辑显示，只有当 open 或 write 失败时，程序才会进入执行 system("warning") 的错误处理分支。在没有写权限限制的情况下，程序会正常写入文件并退出。使用了ulimit -f 0（限制文件写入大小为 0）并配合 trap '' SIGXFSZ（忽略信号导致的崩溃），迫使 write() 函数返回失败。成功诱导程序走向了预设的漏洞分支。

绕过 Shell 的安全保护机制： 现代 Linux 的 Shell（如 Dash）具备安全防护，当检测到有效用户 ID（EUID）与真实用户 ID（UID）不匹配时会自动放弃 Root 权限。在脚本中使用 #!/bin/bash 配合 /bin/bash -p（特权模式标志）成功绕过了这一限制。-p 参数强制 Bash 保留 SUID 带来的 Root 身份，从而让你直接进入具有最高权限的交互式 Shell。
