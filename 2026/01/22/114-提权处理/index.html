第一种是read.sh

核心原理

这个脚本的漏洞在于这一行：

Bash

当 Bash 执行这行代码时，发生了一个关键步骤：参数扩展 (Argument Expansion)。

  1. Shell 会先执行 $(< /root/root.txt)，读取 flag 文件的内容。

  2. 然后，它将读取到的内容作为参数传递给 grep 命令。

  3. 最终执行的命令实际上变成了类似：grep -q "flag{YOUR_FLAG_HERE}"。

    这时我们需要开两个终端

    一个终端运行read.sh

    另一个终端输入

第二种是short.sh

脚本逻辑分析

/opt/short.sh：

  1. My_guess=$RANDOM: 脚本一开始生成了一个 0 到 32767 之间的随机整数。
  2. cat << EOF ... EOF: 这一大段只是打印出来的文本（用来迷惑你的），它并没有实际执行里面的 if 语句。
  3. [ "$1" != "$My_guess" ] && echo "Nop" || bash -i: 这是真正的核心逻辑：
    • 它检查你传入的第一个参数 ($1) 是否不等于刚才生成的随机数 ($My_guess)。
    • 如果不相等（猜错了），输出 “Nop”。
    • 如果相等（猜对了），执行 bash -i。

关键点： bash -i 会启动一个新的交互式 Shell。通常在 CTF 中，这类脚本如果被赋予了 SUID 权限（或者由 root 运行），一旦你猜对数字进入这个新 Shell，你就拥有了 root 权限。

等待它自动跳转到shell
